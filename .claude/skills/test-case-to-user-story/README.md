# Skill: Test Case → User Story

Convert existing test case markdown files into a structured Jira-ready User Story with Acceptance Criteria and test coverage table.

## When to use

- Test cases exist but no parent User Story was ever written
- You need to backfill Jira stories from automated test coverage
- A QA engineer wants to surface test scope to a Product Owner in story format

## Required inputs

1. **Test case `.md` files** — in `test-cases/<type>/<feature>/` (generated by `test-case-generator` skill)
2. **Actor** — the persona the story is written for (e.g. "user", "admin")
3. **Intent** — what they want to do (e.g. "create named canvases of different types")
4. **Business value** — why it matters (e.g. "organise my campaign content")

## Naming rule

**Jira story `summary` = the name of the functionality being tested**, title-cased and matching the spec file name.
Examples: `create-functionality.spec.ts` → `"Create Functionality"`, `login.spec.ts` → `"Login"`.
Never use the user story sentence as the summary.

## Workflow

### 1. Parse test cases

```typescript
import { TestCaseToUserStory } from '.claude/skills/test-case-to-user-story';

const testCases = TestCaseToUserStory.parseTestCasesFromDir('./test-cases');
```

### 2. Build config

```typescript
const config = TestCaseToUserStory.buildConfig(testCases, {
  actor: 'user',
  intent: 'create named canvases of different types',
  businessValue: 'organise my campaign content',
  labels: ['e2e', 'canvas'],
  priority: 'High',
});
```

### 3a. Generate Jira payload and create via MCP

```typescript
const payload = TestCaseToUserStory.generateJiraPayload(config);
// Then pass payload.summary, payload.description, payload.labels, payload.priority
// to mcp__claude_ai_Atlassian_Rovo__createJiraIssue
```

### 3b. Save to markdown file

```typescript
TestCaseToUserStory.saveToFile(config, './test-cases/US-CREATE.md');
```

## Output shape

```
## User Story
As a **user**, I want to **create named canvases** so that **I can organise content**.

## Acceptance Criteria
- **AC-001** — Create button is visible on load _(TC-CREATE-001)_
- **AC-002** — Name input accepts text _(TC-CREATE-002)_
...

## Test Coverage
| Test Case     | Description                        | Priority |
|---------------|------------------------------------|----------|
| TC-CREATE-001 | Create button is visible on load   | High     |
...
```

## Rules

- One AC per test case — no merging, no splitting
- AC IDs are `AC-<NNN>` (3-digit, zero-padded), sequential
- Each AC references its source test case ID in parentheses
- Summary follows: `As a <actor>, I can <intent> so that <value>`
- Never hardcode credentials or environment-specific URLs in the story body
- `labels` should match existing test tags (e.g. `e2e`, module name)

## Jira MCP integration

After generating the payload, create the story with:

```
mcp__claude_ai_Atlassian_Rovo__createJiraIssue
  cloudId: <from getAccessibleAtlassianResources>
  projectKey: <target project>
  issueTypeName: Story
  summary: payload.summary
  description: payload.description
  contentFormat: markdown
  additional_fields: { priority: { name: payload.priority }, labels: payload.labels }
```
